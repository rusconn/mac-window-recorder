# パフォーマンス

## フレームカウントの意味

録画終了時にログ出力される3つの指標:

| 指標 | 意味 |
|------|------|
| 受信フレーム数 | セッション開始後にアプリが処理に入れたフレーム総数 |
| 書き込み | encoder/writerへの書き込みに成功したフレーム数 |
| ドロップ | 受信したが書き込みに至らなかったフレーム数（= 受信 - 書き込み） |

### セッション開始前のフレーム

音声入力がある場合、セッションは最初の音声フレーム到着時に開始される。それ以前の映像フレームは受信・書き込み・ドロップのいずれにもカウントされず、単に破棄される。

## 録画結果のfps60割れ

| 要因 | 実行場所 | 原因 | 制御方法 |
|------|---------|------|-----------|
| コンポジタがフレーム未生成 | OS | GPU/OS負荷によりvsyncサイクルを逃す | なし |
| SCStream内のドロップ（時間） | SCStream（システム） | minimumFrameInterval未満の間隔で配信されたフレームをドロップ | minimumFrameIntervalの設定（`1/120`でほぼ解消） |
| SCStream内のドロップ（バッファ） | SCStream（システム） | アプリによる消費が追いつかず、バッファが満杯になった | encoder/writerの高速化、queueDepthの増大 |
| 不完全フレームの破棄 | アプリ側 | SCFrameStatus != .complete | なし |

## SCStreamのminimumFrameInterval設定

`minimumFrameInterval`は**最小間隔（下限）**であり、最大間隔（上限）ではない。60Hzディスプレイでは`1/120`が最適。

- `1/60`: コンポジタのフレーム間隔が不規則な場合（例: 16ms, 16ms, 16ms, **8ms**, 25ms）、8msのフレームは「1/60秒未満」と見なされSCStreamがドロップ。結果的に大きなギャップ（25ms）が発生し、カクつきの原因となる
- `1/120`: 8msのフレームも配信される。不規則な間隔でもフレームドロップがなくなり、滑らかな出力を実現

### 実測結果

| 設定 | fps | 受信フレーム | ドロップ | 大きな間隔(>20ms) |
|------|-----|------------|---------|-----------------|
| `minimumFrameInterval = 1/60` | ~57fps | 3783 | 51 | 54回 |
| `minimumFrameInterval = 1/120` | **~60fps** | （更多） | （更少） | （ほぼ無し） |

`1/60`で54回の大きな間隔がある = SCStreamがフレームをドロップしてギャップを生んでいる。

## SCStreamのqueueDepth設定

`queueDepth`はSCStream内部のリングバッファ容量。Compositorとアプリの`SCStreamOutput`コールバック間に存在する。

```
Compositor → [SCStream buffer (queueDepth=5)] → コールバック → pendingFrames[] → Writer
```

バッファ満杯時、SCStreamが**システム側でフレームをドロップする**

## 大きな間隔(largeGapCount)

\>20msのフレーム間隔。

大きな間隔が発生する可能性:

- **Compositorのフレーム未生成（OS側）**
- **SCKによる配信間隔の揺れ**
- **アプリ側のフレームドロップ**

区別不可能。`largeGapCount`だけでは原因を特定できない。

## アプリ側キューの効果

pendingFrames（AVAssetWriterパス）
- 役割: SCStreamコールバック（プロデューサ）とAVAssetWriterInput（コンシューマ）の間のバッファ
- 効果
  - SCStreamコールバックの軽量化
  - isReadyForMoreMediaDataが一時的にfalseになってもフレームを保持し、復帰後に再開
  - タイミング揺れの吸収
- 特性: unbounded（上限なし）。コンシューマ遅延時はキューが膨張し、終了時に5秒で強制破棄

semaphore（FFmpegパス）
- 役割: 同時処理数の制限枠
- 効果
  - メモリ消費量の制限
  - semaphore.wait()がコールバックスレッドをブロックし、SCStreamへの背圧を生成
  - タイミング揺れの吸収
- 特性: bounded。ブロック時にSCStreamのバッファが満杯になり、システム側でドロップ

## スタッター検証

`scripts/verify-stutter.sh`でキャプチャ結果のフレーム間隔を自動検証する。初期フレーム（1秒）は除外し、安定状態のスタッター率を計測する。

```
scripts/verify-stutter.sh <video_file> [expected_fps] [skip_seconds]
```

### 診断マトリクス

`verify-stutter.sh`は安定状態のスタッター率で判定する（1%超でWARN、5%超でFAIL）。

| スタッター率 | 視覚確認 | 結論 |
|---|---|---|
| 0% | 滑らか | 正常 |
| 0% | カクつく | **入力元の問題**（キャプチャは正常動作） |
| 高い | カクつく | キャプチャパイプラインの問題 |
| 高い | 滑らか | 不明（要調査） |
